# 开发规范

## 1. 项目结构

当前项目统一使用 `shared + features + sections + content + app` 的分层结构：

```txt
src/
  app/                    # 路由入口、layout、redirect、api
  content/                # 文案、静态内容、i18n 配置
  shared/                 # 全局复用能力
  features/               # 可复用业务功能模块
  sections/               # 页面级区块
```

各层职责必须明确：

- `app/` 只负责路由入口，不写大段页面 UI。
- `content/` 负责文案、页面静态内容、语言字典。
- `shared/` 放跨页面通用能力，如 `ui`、`layout`、`lib`。
- `features/` 放有明确业务含义、可被多个页面复用的模块。
- `sections/` 放页面主体区块，只服务页面组合层。

## 2. 文件夹职责

### `src/app`

只允许放这些内容：

- `layout.tsx`
- `page.tsx`
- `loading.tsx`
- `error.tsx`
- `not-found.tsx`
- `api/*`
- route segment 目录

禁止事项：

- 不要在 `app/page.tsx` 里直接堆复杂 JSX。
- 不要在 `app/*/page.tsx` 里写大量动画、卡片、列表、表单细节。
- 不要在 `app` 层放通用组件。

### `src/shared`

推荐子目录：

```txt
shared/
  ui/                     # 基础 UI 组件
  layout/                 # 页面头部、provider、壳层
  lib/                    # 工具函数、helper、跨业务基础逻辑
  types/                  # 公共类型
```

适合放进 `shared` 的内容：

- 全局 provider、页面壳层、通用页头
- Button、Input 这类基础组件
- 日期格式化、路径拼接、className 工具
- 所有页面都可能用到的布局壳层

### `src/features`

每个 feature 自己成组，按功能命名，例如：

```txt
features/
  navigation/            # 全站导航、顶部栏、下拉导航
  language-switcher/
  search/
  commits/
  daily-record/
```

每个 feature 内建议包含：

- `components/`
- `lib/`
- `types.ts` 或 `model/`

适合放进 `features` 的内容：

- 顶部栏这类带明确业务行为的公共导航
- 搜索框交互
- 语言切换逻辑
- commit 数据读取
- 日常记录卡片
- 导航下拉菜单

### `src/sections`

页面区块按页面或场景命名，例如：

```txt
sections/
  home/
  about/
  daily-record/
  web-log/
  website/
  core/
```

适合放进 `sections` 的内容：

- 首页 Hero
- About 页面主体
- Web Log 页面主体
- Core 页面整段滚动叙事

原则：

- section 可以组合多个 feature 和 shared 组件。
- section 不应该反向被 `shared` 引用。

### `src/content`

当前主要放国际化和静态内容：

```txt
content/
  i18n/
    config.ts
    dictionaries.ts
    index.ts
```

后续如果页面静态内容变多，可以继续拆：

- `content/about.ts`
- `content/daily-record.ts`
- `content/featured.ts`

## 3. 命名规范

- 目录名统一使用 `kebab-case`
- React 组件文件使用 `PascalCase.tsx`
- 普通工具文件使用 `camelCase.ts` 或语义化英文名
- route 目录名必须与 URL 一致，使用英文小写
- 不要使用拼音缩写命名业务目录
- 不要在源码目录中使用中文目录名

示例：

- 正确：`features/language-switcher/components/LanguageSwitcher.tsx`
- 正确：`sections/daily-record/DailyRecordSection.tsx`
- 正确：`features/navigation/components/Navbar.tsx`
- 错误：`components/gonggong/daohang.tsx`
- 错误：`src/组件/顶部栏.tsx`

## 4. 新功能开发规则

### 新增一个新页面

按下面顺序创建：

1. 在 `src/app/[locale]/xxx/page.tsx` 建路由入口
2. 在 `src/sections/xxx/` 建页面主体 section
3. 如果页面里有可复用业务模块，再拆到 `src/features/`
4. 如果只是纯视觉结构，优先留在 `sections/`
5. 文案放到 `src/content/i18n/dictionaries.ts`

页面入口应保持很薄：

```tsx
import type { Metadata } from "next";
import SomeSection from "@/sections/some/SomeSection";
import { getLocaleContext, type LocalePageProps } from "@/shared/lib/locale";

export async function generateMetadata({ params }: LocalePageProps): Promise<Metadata> {
  const { dictionary } = await getLocaleContext(params);
  return dictionary.some.metadata;
}

export default async function SomePage({ params }: LocalePageProps) {
  const { locale, dictionary } = await getLocaleContext(params);
  return <SomeSection locale={locale} dictionary={dictionary.some} />;
}
```

### 新增一个公共组件

判断规则：

- 多个页面都会用：放 `shared/`
- 只有某一类业务会复用：放 `features/`
- 只服务某一个页面结构：放 `sections/`

### 新增一个通用顶部/底部/页头

- 如果只是纯布局壳层，放 `shared/layout/`
- 如果它本身带导航、搜索、切换语言等业务行为，放 `features/`
- 不允许在多个页面各写一份同样的头部
- 站内导航必须统一使用 `getLocalizedPath`

## 5. 国际化规则

- 所有中英文文案统一收口到 `src/content/i18n/dictionaries.ts`
- 不要在组件里硬编码大段中文和英文分支
- 新增英文文案时必须同步补全中文
- 新增中文文案时必须同步补全英文
- 站内链接必须通过 `getLocalizedPath(locale, path)` 生成

禁止事项：

- 组件里直接写 `locale === "zh" ? "中文" : "English"` 的大段内容
- 直接写死 `/about`、`/web-log` 这种不带语言前缀的站内地址

## 6. 依赖方向

依赖关系必须单向：

- `app -> sections -> features -> shared`
- `content` 可被 `app / sections / features / shared` 读取

禁止反向依赖：

- `shared` 不允许引用 `features`
- `shared` 不允许引用 `sections`
- `features` 不允许引用 `sections`
- `content` 不允许依赖页面组件

## 7. 每类文件建议包含什么

### `shared/ui`

适合包含：

- `button.tsx`
- `input.tsx`
- `modal.tsx`
- `tooltip.tsx`

要求：

- 无业务语义
- 可纯 props 驱动
- 样式和交互尽量通用

### `feature/components`

适合包含：

- 某业务自己的 UI 组件
- 可被多个页面复用的交互块

要求：

- 文件名体现业务含义
- 可以依赖 `shared`
- 可以依赖同目录 `lib`

### `sections`

适合包含：

- 页面主视觉区块
- 页面布局容器
- 页面内容编排

要求：

- section 负责“组合”
- 尽量不要把纯业务逻辑塞进 section
- 复杂逻辑应下沉到 feature 或 shared

## 8. AI 开发执行规则

以后让 AI 改项目时，默认遵循下面规则：

1. 先判断改动属于 `shared`、`features` 还是 `sections`
2. 页面入口保持最薄，不要直接在 `app` 里堆实现
3. 优先复用已有公共组件，不重复造轮子
4. 新文案必须补齐中英文
5. 新站内链接必须带 locale
6. 新目录与文件统一英文命名
7. 同类文件放进同类目录，不允许随手平铺到 `src/components`

## 9. 提交前检查

每次新增功能或重构后，至少检查：

```bash
npm run lint
npm run build
```

同时手动确认：

- 顶部栏是否仍可用
- 语言切换是否正常
- 站内链接是否保留 `/zh`、`/us`
- 文案是否中英文完整
- 新文件是否放到了正确分层
