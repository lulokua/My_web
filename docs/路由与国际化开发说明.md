# 路由与国际化开发说明

本文档说明当前项目的语言路由结构、相关配置文件作用，以及后续新增页面或功能时需要遵循的开发约定。

当前项目使用 Next.js App Router，并通过 URL 第一段作为语言前缀：

- `/us`：英文站点
- `/zh`：中文站点
- `/us/web-log`：英文网站日志
- `/zh/web-log`：中文网站日志

注意：当前代码实际使用的是 `/zh`，不是 `/ch`。如果后续必须使用 `/ch`，需要统一调整 `src/lib/i18n.ts` 中的语言配置，并同步修改相关路径和文案键。

## 当前路由组织结构

核心路由位于 `src/app`：

```txt
src/app
├─ layout.tsx
├─ page.tsx
├─ web-log/
│  └─ page.tsx
└─ [locale]/
   ├─ layout.tsx
   ├─ page.tsx
   └─ web-log/
      └─ page.tsx
```

各文件职责如下：

- `src/app/layout.tsx`
  - 全站根布局。
  - 只负责加载全局 CSS 和渲染 `children`。
  - 不直接放置 `Navbar`，因为导航栏需要跟随语言变化。

- `src/app/page.tsx`
  - 根路径 `/` 的入口。
  - 当前会重定向到默认语言 `/${defaultLocale}`，也就是 `/us`。

- `src/app/web-log/page.tsx`
  - 旧路径 `/web-log` 的兼容入口。
  - 当前会重定向到 `/us/web-log`。

- `src/app/[locale]/layout.tsx`
  - 语言路由布局。
  - 校验 `locale` 是否为受支持语言。
  - 根据语言加载字典，并渲染对应语言的 `Navbar`。

- `src/app/[locale]/page.tsx`
  - 语言首页。
  - 例如 `/us` 和 `/zh`。
  - 页面文案从 `src/lib/i18n.ts` 的字典读取。

- `src/app/[locale]/web-log/page.tsx`
  - 语言化网站日志页面。
  - 例如 `/us/web-log` 和 `/zh/web-log`。
  - 页面标题、返回按钮、错误提示、日期格式等均按语言处理。

## 语言配置

语言相关配置集中在 `src/lib/i18n.ts`。

当前主要导出：

```ts
export const locales = ["zh", "us"] as const;
export type Locale = (typeof locales)[number];
export const defaultLocale: Locale = "us";
```

重要约定：

- 所有支持的语言必须写入 `locales`。
- 默认语言由 `defaultLocale` 控制。
- URL 路径拼接统一使用 `getLocalizedPath(locale, path)`。
- 页面和组件文案统一维护在 `dictionaries` 中。
- 不要在页面组件里散落硬编码中英文文案。

示例：

```ts
getLocalizedPath("us"); // /us
getLocalizedPath("zh"); // /zh
getLocalizedPath("us", "/web-log"); // /us/web-log
getLocalizedPath("zh", "/web-log"); // /zh/web-log
```

## Middleware 路由规则

路径语言识别和默认跳转由 `src/middleware.ts` 处理。

当前规则：

- 如果路径第一段是 `zh` 或 `us`，直接放行。
- 如果路径没有语言前缀，则自动补默认语言 `/us`。
- `_next`、`api` 和静态资源路径不参与语言跳转。
- 逻辑只判断路径，不绑定具体域名。

例如：

```txt
/              -> /us
/web-log       -> /us/web-log
/zh            -> 放行
/us            -> 放行
/zh/web-log    -> 放行
/us/web-log    -> 放行
```

这意味着部署到任何域名时，只要路径带上 `/zh` 或 `/us`，都会进入对应语言站点。

## 新增路由规范

新增普通页面时，应优先放在 `src/app/[locale]` 下。

例如新增 About 页面：

```txt
src/app/[locale]/about/page.tsx
```

页面路径会自动变成：

```txt
/us/about
/zh/about
```

推荐页面结构：

```tsx
import type { Metadata } from "next";
import { notFound } from "next/navigation";

import { getDictionary, isLocale, type Locale } from "@/lib/i18n";

type PageProps = {
  params: Promise<{ locale: string }>;
};

async function resolveLocale(params: PageProps["params"]): Promise<Locale> {
  const { locale } = await params;

  if (!isLocale(locale)) {
    notFound();
  }

  return locale;
}

export async function generateMetadata({ params }: PageProps): Promise<Metadata> {
  const locale = await resolveLocale(params);
  const dictionary = getDictionary(locale);

  return {
    title: dictionary.metadata.title,
    description: dictionary.metadata.description,
  };
}

export default async function AboutPage({ params }: PageProps) {
  const locale = await resolveLocale(params);
  const dictionary = getDictionary(locale);

  return <main>{dictionary.nav.links.home}</main>;
}
```

如果需要保留旧路径，例如 `/about`，可以新增兼容重定向页面：

```txt
src/app/about/page.tsx
```

内容示例：

```tsx
import { redirect } from "next/navigation";

import { defaultLocale } from "@/lib/i18n";

export default function AboutRedirect() {
  redirect(`/${defaultLocale}/about`);
}
```

## 新增文案规范

所有跨语言文案都应写入 `src/lib/i18n.ts` 的 `dictionaries`。

示例：

```ts
export const dictionaries = {
  us: {
    about: {
      title: "About",
    },
  },
  zh: {
    about: {
      title: "关于",
    },
  },
} as const;
```

约定：

- 新增英文文案时，必须同步补齐中文文案。
- 新增中文文案时，必须同步补齐英文文案。
- 组件不要自行判断 `locale` 后硬编码大段文案。
- 少量格式化逻辑可以根据 `locale` 分支，例如时间文案。

## 组件开发规范

导航和搜索框当前由语言布局统一传入文案。

相关组件：

- `src/components/Navbar.tsx`
- `src/components/ExploreDropdown.tsx`
- `src/components/SearchBox.tsx`

当前数据流：

```txt
src/app/[locale]/layout.tsx
-> getDictionary(locale)
-> <Navbar locale={locale} />
-> Navbar 读取字典
-> ExploreDropdown / SearchBox 接收文案 props
```

开发约定：

- 需要跟随语言变化的组件，应通过 props 接收文案或语言。
- 客户端组件可以接收 `locale`，但不要在客户端重复维护整份字典。
- 内部链接必须使用 `getLocalizedPath(locale, path)` 生成。
- 不要写死 `/web-log`、`/about` 这类不带语言前缀的站内链接。

正确示例：

```ts
getLocalizedPath(locale, "/web-log");
```

避免：

```ts
"/web-log";
```

## 相关配置文件作用

### `src/lib/i18n.ts`

集中维护：

- 支持的语言列表。
- 默认语言。
- 语言校验函数。
- 本地化路径生成函数。
- 中英文文案字典。

后续新增语言、修改默认语言、新增文案，优先修改此文件。

### `src/middleware.ts`

负责无语言前缀路径的自动跳转。

此文件不应该写死域名，只处理路径。这样本地开发、测试域名和正式域名都能使用同一套规则。

### `src/app/[locale]/layout.tsx`

负责语言层级布局：

- 校验语言参数。
- 加载语言字典。
- 渲染导航栏。
- 为语言内容区域设置 `lang`。

### `src/lib/commits.ts`

负责读取和格式化 GitHub commit 缓存数据。

`formatCommitDate(value, locale)` 会根据语言格式化日期：

- `zh` 使用 `zh-CN`
- `us` 使用 `en-US`

### `next.config.ts`

当前主要配置：

- `allowedDevOrigins`：允许本地开发来源。
- `images.remotePatterns`：配置 `next/image` 可加载的远程图片域名。

如果新功能使用了新的远程图片域名，需要同步添加到 `images.remotePatterns`。

## 开发新功能注意事项

新增页面时：

- 页面放到 `src/app/[locale]/...` 下。
- 所有站内链接带语言前缀。
- 页面入口先校验 `locale`。
- 页面元数据使用 `generateMetadata` 并从字典读取。
- 需要旧路径兼容时，在非语言路径下新增 redirect 页面。

新增组件时：

- 展示文案从上层传入。
- 站内链接由上层或组件内通过 `getLocalizedPath` 生成。
- 如果组件是 client component，不要读取服务端专用 API。

新增语言时：

- 修改 `locales`。
- 为 `dictionaries` 增加完整语言字典。
- 检查 `middleware` 是否需要调整默认跳转或别名。
- 检查日期、时间、数字格式化逻辑。
- 运行 `npm run build` 验证所有静态路径。

修改中文路径为 `/ch` 时：

- 当前项目使用 `/zh`。
- 如果必须改为 `/ch`，建议先决定是“替换”还是“兼容别名”。
- 替换方式：将 `locales = ["zh", "us"]` 改为 `["ch", "us"]`，并把字典键 `zh` 改为 `ch`。
- 兼容方式：保留 `zh`，在 `middleware` 或专门 redirect 页面里把 `/ch` 跳到 `/zh`，或反过来。
- 修改后必须检查所有已有链接、文档和部署规则。

## 验证方式

修改路由或文案后，至少执行：

```bash
npm run build
```

推荐手动检查：

```txt
http://127.0.0.1:3001/us
http://127.0.0.1:3001/zh
http://127.0.0.1:3001/us/web-log
http://127.0.0.1:3001/zh/web-log
```

如果访问 `/` 或旧路径 `/web-log`，应自动跳转到默认英文路径。
